    MySql-索引基础使用

    是秃子总会发光 Mysql

    # 简述

    • 索引是一种有序的数据结构。通过空间换时间

    • 索引可以减少磁盘IO、可以降低排序消耗

    • 索引占用磁盘空间(可忽略影响，在isd、idb等文件中存储)，降低了数据更新效率(需要更新所有结构)

    • 索引在存储层完成，不同的存储引擎的索引结构不同

    SELECT * FROM users WHERE age=45
    1

    没有索引：全表扫描，即便已经找到了，还是会扫描表中的每一条数据

    有索引：右侧是索引结构(这里只是演示)，查找36->48->45共三次，效率高

    # 索引结构

    不指定则默认使用的是B+Tree索引

    索引类型 描述
    B+Tree索引 最常见的索引类型，大部分引擎都支持B+树索引
    Hash索引 底层数据结构是用哈希表实现的，只有精确匹配索引列的查询才有效，不支持范围查询
    R-tree(空间索引) 空间索引是MyISAM引擎的一个特殊索引类型，主要用于地理空间数据类型，通常使用较少
    Full-text(全文索引) 是一种通过建立倒排索引，快速匹配文档的方式。类似于Lucene,Solr,ES
    索引类型 InnoDB MyISAM Memory
    B+tree索引 支持 支持 支持
    Hash索引 不支持 不支持 支持
    R-tree索引 不支持 支持 不支持
    Full-text 5.6版本之后支持 支持 不支持

    # 数据结构

    # 二叉树

    • 顺序插入时(右侧类型)，实际是链表，查询效率低(查找元素，需要遍历)
    • 每个节点最多两个数据，数据量大时高度大，检索慢

    # 红黑树

    红黑树可以解决第一个不平衡的问题，但是它也是二叉树，还是存在树高度大的问题

    # B-Tree

    • B树(多路平衡查找树)
    • 下方以一颗最大度数(即一个节点下的节点数)为3的b-tree为例(该节点最多有2个数)
    • 具体可见可视化网站数据结构可视化 (opens new window)

    • B树解决了平衡和树深度的问题
    • 每个点也是键和值绑在一起

    # B+Tree

    B树的升级

    特性 B树 B+树
    数据存储方式 键值和数据都存储在内部节点和叶子节点上 数据只存储在叶子节点，内部节点只存储键值（所有数据都会出现在叶子节点）
    叶子节点的链接 叶子节点是独立的，没有链接 所有的叶子节点都通过指针相互链接，形成一个链表结构
    搜索效率 一般 由于所有数据都在叶子节点，且叶子节点形成链表，因此搜索效率更高
    插入和删除操作 效率一般 由于结构更简单，插入和删除操作通常更快

    # B+Tree索引

    • Mysql的B+树叶子节点是双向循环链表
    • 这个特性有助于提高范围查询的性能，有助于叶子节点(数据都在这)的遍历

    # hash索引

    # 概述

    • 哈希索引就是采用一定的hash算法，将键值换算成新的hash值，映射到对应的槽位上，然后存储在hash表中。

    • 如果两个（或多个）键值，映射到一个相同的槽位上，他们就产生了hash冲突（也称为hash碰撞），可以通过链表来解决。

    • 在MySQL中，支持hash索引的是Memory引擎，而InnoDB中具有自适应hash功能，hash索引是存储引擎根据B+Tree索引在指定条件下自动构 建的。

    # 特点

    • Hash索引只能用于对等比较(=，in),不支持范围查询(between,>,<,…)
    • 无法利用索引完成排序操作(无效)
    • 查询效率高，通常只需要一次检索就可以了(不出现hash冲突)，效率通常要高于B+tree索引

    # 索引分类

    # 简介

    默认使用InnoDB，主键索引的结构是B+Tree

    分类 含义 特点 关键字
    主键索引 针对于表中主键创建的索引 默认自动创建，只能有一个 PRIMARY
    唯一索引 避免同一个表中某数据列中的值重复 可以有多个 UNIQUE
    常规索引 快速定位特定数据 可以有多个 无
    全文索引 全文索引查找的是文本中的关键词，而不是比较索引中的值 可以有多个 FULLTEXT

    在InnoDB中，根据索引存储形式，还可以分为：

    分类 含义 特点
    聚集索引(Clustered Index) 将数据存储与索引放到了一块，索引结构的叶子节点保存了行数据 必须有而且只有一个
    二级索引(Secondary Index) 将数据与索引分开存储，索引结构的叶子节点关联的是对应的主键 可以存在多个

    注意

    • 有主键，主键索引就是聚集索引
    • 没有主键，第一个唯一索引就是聚集索引
    • 否则字段生成一个rowid为隐藏聚集索引
    • 其它就是二级索引(辅助索引)

    # 查询过程(回表查询)

    在InnoDB存储引擎中,不按主键查找

    有二级索引:

    • 查询二级索引获取主键id,如果找不到，停止查询并返回空

    • 使用主键值在聚簇索引中定位到完整的数据行

    只有聚簇索引:

    • 直接查询聚簇索引
    -- 这里没有直接查找主键
select from user where name='Arm'
    1
    2

    以下SQL语句，哪个执行效率高？为什么？ 备注：id为主键，name字段创建的有索引；

    select from user where id =10;
select from user where name='Arm';

    2

    第一条高，直接走聚集索引，第二条走回表查询。

    # 创建索引

    • 可以为要查询的任意字段创建索引
    • 主键索引和唯一索引需要字段值不能重复
    • 主键索引只能有一个，会默认创建,不需要我们创建
    -- 索引可以关联一个(单列索引)或多个(联合、复合索引)表字段，默认创建的是常规索引(快速定位特定数据)
CREATE [UNIQUE | FULLTEXT] INDEX index_name ON table_name (index_col_name,...);
